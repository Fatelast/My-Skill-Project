# Miniprogram Prototype Designer

## Overview

这个 skill 用于把产品需求文字快速整理成前端可执行的原型输出，重点是页面结构、流程关系、原型交互、组件拆分、工程结构建议和开发 Prompt。

默认优先面向 `Taro + React` 的微信小程序项目，其次支持 `uni-app`、小程序原生、App、H5 和其他移动端前端产品。输出优先使用中文，最终可交付产物默认优先为 `HTML` 交互原型。

## 何时使用

当用户出现以下诉求时使用本 skill：

- 根据需求文字生成页面原型
- 根据 PRD 拆解页面结构
- 只有功能描述，没有 UI 设计稿
- 希望把模糊想法整理成页面清单、流程图、线框图
- 希望得到可点击、可切换页面的交互原型文件
- 希望得到可直接交给开发 Agent 的实现 Prompt
- 希望得到更贴近 Taro / React / uni-app 的页面和工程拆分规则
- 希望把小程序、移动端、H5 功能拆成页面和组件

典型触发语：

- “根据这个需求生成小程序原型图”
- “帮我把 PRD 转成低保真线框图”
- “根据需求生成可交互原型”
- “输出 HTML 原型文件”
- “把这个需求拆成页面结构和组件”
- “生成页面原型和开发 Prompt”

## 不适用场景

以下情况不要把本 skill 当成最终方案：

- 用户要的是高保真视觉稿、运营海报或品牌视觉设计
- 用户要的是 Figma 可编辑设计稿
- 用户要的是直接生成前端代码，而不是先生成原型与结构

遇到这些场景时，可以把本 skill 作为前置分析工具，但不要把输出伪装成精美 UI 稿。

## 默认原则

- 默认产品类型：微信小程序
- 默认输出语言：中文
- 默认原型类型：`HTML` 交互原型；如果用户只要求结构分析，再输出低保真文本线框图
- 默认技术表达：TypeScript 类型定义
- 默认实现目标：`Taro + React + TypeScript`
- 次选实现目标：`uni-app + Vue 3 + TypeScript`
- 默认目标：结构清晰、便于开发、避免过度发挥
- 若需求不完整：允许合理补全，但必须显式列出“假设”
- 默认交付文件：`index.html`、`pages/*.html`、`assets/*.css`、`assets/*.js`
- 默认输出目录：项目根目录下的 `prototype-output/`
- 文件边界规则：不得在原有业务目录、源码目录、配置目录中新增或修改文件，只能在项目根目录新建独立原型目录承接输出

## 工作流

按下面顺序工作，不要跳步。

### Step 1: 理解输入

先从用户输入中提取：

- 产品类型
- 目标用户
- 核心目标
- 功能模块
- 关键业务流程
- 关键页面对象
- 关键数据字段

如果用户没有明确说明产品类型，但语境明显偏移动端，默认按微信小程序处理；如果也没有指定前端框架，默认按 `Taro + React` 组织输出。

### Step 2: 补充假设

当需求缺少关键信息时，先补充最小必要假设，例如：

- 是否需要登录
- 是否存在订单/记录页
- 是否存在详情页
- 列表是否支持筛选
- 提交后是否有成功页

假设必须保守，不能无依据扩展复杂功能。

### Step 3: 识别页面与流程

把需求拆成页面层级和核心路径，优先识别：

- 首页
- 列表页
- 详情页
- 表单/提交页
- 订单/记录页
- 我的/个人中心
- 结果页

然后输出页面跳转关系，优先使用 Mermaid `flowchart TD`。

### Step 4: 先生成结构化原型

先完成页面层级、流程与关键交互梳理，再决定交付形式。

如果用户只要分析稿或线框稿：

- 输出 ASCII 或 Unicode 文本线框图
- 用于快速评审信息架构和页面模块

如果用户要最终原型文件：

- 默认输出 `HTML` 交互原型
- 原型至少支持页面跳转、Tab 切换、弹层开关、筛选确认、表单填写与提交反馈
- 不要只输出说明文档，必须输出可打开的文件结构
- 原型文件必须落在项目根目录新建的独立目录中，默认使用 `prototype-output/`
- 不得把原型页面、样式、脚本直接写入现有 `src/`、`pages/`、`components/`、`config/` 等业务目录

### Step 5: 生成交互原型文件

默认交付 `HTML` 原型，并遵守以下规则：

- 所有页面都能通过浏览器直接打开
- 支持基础点击交互，不依赖后端接口
- 用本地 mock 数据驱动页面内容
- 只允许在项目根目录下创建一个独立原型目录
- 样式集中到 `assets/*.css`
- 交互集中到 `assets/*.js`
- 页面数量较多时，允许拆分为多页 HTML
- 必须提供一个总入口页，或默认首页入口
- 不得修改现有项目业务文件，除非用户明确取消该限制

若用户明确要求其他格式：

- `.sketch`：仅在用户明确要求且本地工具链可稳定生成时尝试，否则回退到 `HTML`
- `.mp`：仅在用户明确说明该格式对应的具体工具时尝试，否则回退到 `HTML`

不要伪造无法稳定生成的专有格式文件。

### Step 6: 补充页面说明

每个核心页面至少说明：

- 页面目标
- 页面模块
- 关键字段
- 主操作
- 次操作
- 加载态
- 空状态
- 错误态

### Step 7: 输出开发导向内容

最后补充面向实现的开发信息：

- 组件拆分建议
- 基础数据结构建议
- 页面目录与路由建议
- hooks / store / services 边界建议
- 关键接口字段占位建议
- 可直接交给开发 Agent 的实现 Prompt

如果用户未指定技术栈，默认用 `Taro + React + TypeScript` 作为 Prompt 目标，并用 TypeScript 形式表达数据结构。

### Step 8: 交付原型文件清单

当输出 `HTML` 原型时，最后必须明确：

- 原型根目录名称
- 原型目录结构
- 入口文件
- 页面文件
- 样式文件
- 脚本文件
- 如何打开与预览

## 输出顺序

默认按以下顺序输出，除非用户明确要求裁剪：

1. 需求理解
2. 假设
3. 页面清单
4. 用户流程
5. 原型输出边界说明
6. 页面详细说明
7. 原型交互说明
8. 原型文件结构建议
9. 低保真线框图（当需要时）
10. 组件拆分建议
11. 页面目录与路由建议
12. 状态、接口与复用边界建议
13. 数据结构建议
14. 开发 Agent Prompt

输出时优先参考 [references/prototype-output-template.md](references/prototype-output-template.md) 的章节结构和格式。

## 输出要求

- 结构清晰，标题明确
- 面向开发可落地，不停留在产品分析
- 页面、模块、字段、操作都要具体
- 若目标是最终原型文件，优先输出可运行的 `HTML` 原型而不是只写线框说明
- 若需求模糊，先收敛信息架构，再画线框
- 若页面较多，先覆盖主流程页面，再补次级页面
- 若目标是 Taro / React / uni-app，要显式体现页面文件组织、组件边界和跨端限制
- Prompt 中优先要求输出可直接落地的目录结构、页面文件、组件文件、mock 数据和接口占位
- 不要声称生成了 `.sketch` 或 `.mp`，除非实际知道对应工具格式并且有稳定生成路径
- 必须明确说明原型文件只输出到项目根目录新建的独立目录
- 不得修改、覆盖、混入原项目现有业务目录

## 技术栈适配

如果用户指定技术栈，按用户要求调整输出中的术语、目录建议和 Prompt：

- 微信小程序原生
- Taro
- uni-app
- Vue
- React
- HTML 原型
- Figma 结构说明

如果用户没有指定，不要主动切到桌面端或复杂多端架构；默认优先产出以下风格：

- `Taro`：页面文件按 `pages/<page>/index.tsx`、`index.config.ts`、`index.scss` 思路组织
- `React`：强调函数组件、hooks、局部状态与可复用业务组件
- `uni-app`：页面按 `pages.json`、`pages/<page>.vue` 或模块化子目录组织，注意小程序兼容能力
- `HTML 原型`：页面按 `index.html` + `pages/*.html` + `assets/*.css/js` 组织，支持点击跳转与基础交互

在 `Taro + React` 场景下，输出时优先补充：

- 页面路由清单
- 页面目录结构
- 组件目录结构
- hooks、store、services、constants 的职责边界
- 页面级 mock 数据和接口占位
- 页面配置项建议，例如导航标题、下拉刷新、分享配置占位

在 `uni-app` 场景下，输出时优先补充：

- `pages.json` 页面注册建议
- 页面与分包建议
- Vue 组件拆分
- composables 或公共方法拆分建议
- API 模块、状态模块、样式变量组织建议

在 `HTML 原型` 场景下，输出时优先补充：

- 项目根目录独立输出目录名称
- 多页面 HTML 目录结构
- 全局样式与页面样式拆分
- 基础交互脚本
- 页面跳转映射表
- 模拟数据文件或内嵌 mock 数据策略
- 浏览器直接打开的预览说明

## 质量检查

输出前自查：

- 页面是否覆盖主流程
- 是否明确列出了假设
- 流程图是否能闭环
- 每个核心页面是否有结构化线框或对应 HTML 页面
- 组件拆分是否支持复用
- 工程结构是否能映射到 Taro / React / uni-app 实际目录
- 状态和接口边界是否足够明确，便于开发 Agent 继续实现
- 开发 Prompt 是否能直接给下游 Agent 使用
- 如果交付的是 HTML 原型，入口文件和页面跳转是否完整可用
- 是否严格把原型文件限制在项目根目录独立目录中
- 是否避免了对现有业务目录的新增和修改

## 失败保护

如果用户给的信息极少，不要直接拒绝。先基于最小假设输出一个可讨论版本，并明确指出：

- 哪些是来自原始需求
- 哪些是基于常见小程序模式的假设
- 哪些点需要用户二次确认
